# 365-User AD and Cloud Setup

A production-ready Python GUI tool for MSP and enterprise admins that automates end-to-end user provisioning:

1. **Create** a user in on-prem Active Directory
2. **Trigger** Entra ID (Azure AD Connect) delta sync
3. **Assign** Microsoft 365 licenses via Microsoft Graph
4. **Manage** both on-prem AD groups and cloud-only Entra ID groups

Runs on a Windows Domain Controller under a domain admin context. **One universal EXE works for any customer** — per-customer settings live in `config.json` next to the EXE.

## Features

- Tkinter GUI — no web server, no browser required
- **First-run Setup Wizard** — walks through customer configuration step by step
- **DPAPI-encrypted certificates** — private keys are protected by Windows, useless if stolen
- External `config.json` — one EXE build, deploy to any customer
- Auto-detects Entra Connect sync server (local, SCP, or DC scan)
- Separate AD group and cloud-only Entra ID group assignment
- License seat availability check with refresh (blocks provisioning if no seats available)
- Password generation with complexity validation
- Manager search via AD lookup
- Auto-generated display name and username (editable)
- Preflight checks with detailed remediation instructions
- Full audit logging to `C:\Logs\UserProvisioning.log`

## Customer Deployment (Quick Start)

For each new customer:

1. **Copy** `UserProvisioning.exe` to the customer's Domain Controller
2. **Run it** — the preflight will detect that `config.json` is missing
3. **Click "Run Setup Wizard"** — it will:
   - Auto-detect the AD domain and NetBIOS name
   - Walk you through creating an App Registration in the customer's Entra ID
   - Generate a certificate and DPAPI-encrypt the private key
   - Save everything to `config.json`
4. **Upload the `.cer` file** to the App Registration in Azure Portal
5. **Grant admin consent** on the API permissions
6. **Click "Retry"** in the preflight — all checks should pass
7. **Click "Continue"** to start provisioning users

### What gets deployed per customer

```
C:\Tools\UserProvisioning\
  UserProvisioning.exe        # Same EXE for every customer
  config.json                 # Customer-specific settings (auto-generated by wizard)
  C:\Certs\
    graph_app.pem.protected   # DPAPI-encrypted private key (machine-bound)
    graph_app.cer             # Public key (uploaded to Azure)
```

## Security Model

| Component | Protection |
|---|---|
| config.json | Non-secret values only (tenant ID, client ID are public identifiers) |
| PEM private key | DPAPI-encrypted — bound to the Windows user account + machine. Stolen file is useless elsewhere |
| Graph access token | In-memory only, never written to disk |
| Passwords | Passed via `-EncodedCommand` (base64), never logged |
| App Registration | Certificate-based auth, no client secrets |

**Recommendations:**
- Restrict NTFS permissions on `config.json` and `.pem.protected` to the service account
- Use minimum required Graph API permissions
- Set certificate expiry to 1-2 years and rotate

## config.json Reference

Created automatically by the Setup Wizard, or manually:

```json
{
  "ad_domain": "customer.com",
  "ad_netbios": "CUSTOMER",
  "email_domains": ["customer.com"],
  "adsync_server": null,
  "graph_tenant_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "graph_client_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "graph_cert_thumbprint": "AABBCCDD...",
  "graph_cert_path": "C:\\Certs\\graph_app.pem.protected",
  "password_min_length": 12,
  "entra_poll_interval_seconds": 15,
  "entra_poll_timeout_seconds": 300
}
```

| Key | Required | Description |
|---|---|---|
| `ad_domain` | Yes | Customer's AD domain FQDN |
| `ad_netbios` | Yes | NetBIOS domain name |
| `email_domains` | Yes | List of email domain options |
| `graph_tenant_id` | Yes | Entra ID Directory (tenant) ID |
| `graph_client_id` | Yes | App Registration Application (client) ID |
| `graph_cert_thumbprint` | Yes | Certificate thumbprint |
| `graph_cert_path` | Yes | Path to DPAPI-encrypted PEM file |
| `adsync_server` | No | Override sync server (null = auto-detect) |
| `password_min_length` | No | Default: 12 |
| `entra_poll_interval_seconds` | No | Default: 15 |
| `entra_poll_timeout_seconds` | No | Default: 300 |

## Prerequisites

### On the Domain Controller
- Windows Server with Active Directory role
- PowerShell with `ActiveDirectory` module (default on DCs)
- Azure AD Connect / Entra Connect (on this DC or another server)
- Outbound HTTPS (443) to `login.microsoftonline.com` and `graph.microsoft.com`

### Azure / Entra ID (per customer)

Create an **App Registration** with these **Application** permissions (grant admin consent):

| Permission | Purpose |
|---|---|
| `User.ReadWrite.All` | Read users, assign licenses |
| `Directory.ReadWrite.All` | Modify license assignments |
| `Organization.Read.All` | Read subscribed SKUs |
| `Group.ReadWrite.All` | Read cloud groups |
| `GroupMember.ReadWrite.All` | Add users to cloud groups |

Upload the `.cer` public key to **Certificates & secrets**.

## Build to EXE

```batch
pip install pyinstaller msal requests
pyinstaller --onefile --windowed --name "UserProvisioning" provision_user.py
```

The resulting `dist\UserProvisioning.exe` is a standalone executable. No Python installation needed on the DC. The GitHub Action builds this automatically on push to `main`.

## Usage

### Run from Source (development)
```
pip install msal requests
python provision_user.py
```

### Run as EXE (production)
```
UserProvisioning.exe
```

First run will show the Setup Wizard. Subsequent runs go straight to preflight checks.

## Known Limitations

- **No rollback**: If AD user creation succeeds but a later step fails, the user is not auto-deleted. Fix the issue and retry manually.
- **Single license per run**: Assigns one license per provisioning. Use M365 admin center for additional licenses.
- **Cloud group pagination**: Fetches up to 999 cloud groups.
- **Entra sync timing**: Delta sync typically takes 30s-3min. Default poll timeout is 5 minutes (configurable in `config.json`).
- **DPAPI scope**: The encrypted cert is bound to the user account that ran the Setup Wizard. If a different admin account needs to run the tool, re-run the wizard to re-encrypt under their account.
